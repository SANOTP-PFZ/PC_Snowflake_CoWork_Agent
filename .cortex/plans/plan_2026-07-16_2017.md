## Frontend Redesign Plan — If I Were Building From Scratch

### Current Problems (Root Causes)

1. **Navigation hack hell** — The core issue is Streamlit's limitation: HTML elements can't trigger `session_state` changes. This led to invisible button overlays, pointer-events hacks, and broken hover effects. The `* { transition: all 0.15s ease; }` rule also causes flicker on every state change.

2. **Inconsistent card system** — Agent cards use `<a href>` (native HTML links, hover works), but category cards need `st.button` for internal navigation. Two different mechanisms = two different behaviors.

3. **CSS bloat** — 370+ lines of CSS with dead selectors (`.agent-card-header`, `.agent-icon`, `.agent-name`, `.agent-chips`, `.chip-*`, `.nav-btn-hidden`, `.brand-card`), overly specific selectors, and hacks targeting Streamlit internals that break between versions.

4. **No visual hierarchy on landing page** — Brand summary + heading + category cards are all at the same visual weight. No clear "hero" section.

5. **Monolithic file structure** — All CSS in one 400-line `st.markdown`, all routing in one file.

### Redesign Plan

#### Step 1: Fix Navigation Architecture (the core fix)

**Use Streamlit's native `st.navigation` / multipage app** if version allows (1.35+), OR use a clean query-param approach that doesn't cause full reloads:

- Keep `session_state["screen"]` routing (it works, is seamless)
- For category cards: Use `st.button` BUT style it as the ENTIRE card using a **wrapper div approach that actually works**:

  **The key insight**: Instead of overlaying an invisible button on top of HTML, put the button FIRST and the visual card AFTER, then use `margin-top: negative` to pull the card UP over the button. This way:
  - User sees the card (HTML with hover)
  - Click passes THROUGH the card to the button underneath (using `pointer-events: none` on the card HTML)
  - Hover works on the card because `pointer-events: none` still allows `:hover` on the element itself via CSS

```css
.category-card-wrapper {
    pointer-events: none;  /* clicks pass through */
    margin-top: -200px;
    position: relative;
}
.category-card-wrapper:hover .ta-agent-card {
    transform: translateY(-4px);
    box-shadow: var(--shadow-lg);
}
```

The button sits BEHIND at full height, the card visual sits ON TOP with `pointer-events: none`.

**Wait — actually `pointer-events: none` DOES prevent `:hover`**. So this won't work either.

#### REVISED Step 4: The Actually Working Approach

The ONLY reliable way to have both hover AND click in Streamlit:

1. Use `st.button` styled AS the card itself (no separate HTML)
2. Style the button with custom CSS to look like a card (large padding, left-align, multi-line via `white-space: normal`)
3. Add a `st.caption` below for description

OR: Accept that Streamlit has this limitation and use `st.columns` with the button taking full width, styled to LOOK like a card. Description goes in a separate element but visually appears inside due to CSS tricks (negative margin, shared background).

**ACTUAL RECOMMENDED**: Keep the current invisible-button-overlay approach BUT fix the hover by using a JavaScript `onmouseover`/`onmouseout` inline style change (Streamlit doesn't strip inline event handlers on divs):

```html
<div class="ta-agent-card category-card" 
     onmouseover="this.style.transform='translateY(-4px) scale(1.02)';this.style.boxShadow='0 12px 40px rgba(0,47,108,0.1)'"
     onmouseout="this.style.transform='';this.style.boxShadow=''">
```

This gives native hover effects regardless of button overlays.
