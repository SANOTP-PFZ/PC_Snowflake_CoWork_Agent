### IMPLEMENTATION PLAN

#### Step 1: Fix Category Cards — Make Them Proper Clickable Cards (P0 #1)

**The core problem:** Streamlit `st.button` cannot render multi-line HTML content and is always styled as a small pill by our global CSS. `st.container(border=True)` is not clickable.

**Solution:** Use the SAME approach as agent cards — render the entire category card as a styled HTML `<div>` with the `.ta-agent-card` class (which already has the curved-edge box styling), and place a HIDDEN `st.button` below it for the actual click target. Hide the button with CSS using a wrapper class.

Implementation:
- In `landing.py`, render each category as:
  ```python
  st.markdown("""<div class="ta-agent-card" style="min-height:120px">
      <div class="ta-agent-name">Therapy Area</div>
      <div style="font-size:13px;color:var(--text-3);margin-top:8px">Description...</div>
  </div>""", unsafe_allow_html=True)
  ```
- Immediately below, render `st.button` with a wrapper div class `category-nav-btn`
- Add CSS: `.category-nav-btn .stButton > button { opacity:0; height:0; padding:0; margin-top:-8px; overflow:hidden; }` — This makes the button invisible but still catches focus/click through Streamlit's rerun cycle
  
**ALTERNATIVE (Recommended):** Use Streamlit's `st.page_link` if available in v1.35+, OR use `st_click_detector` third-party component. But since this deploys on Dataiku and we want minimal dependencies, the hidden-button approach is most reliable.

**BEST ALTERNATIVE (Simplest):** Style the `st.button` ITSELF to look like a card by overriding the global CSS specifically for these buttons using a wrapper div with a custom class. This avoids the hidden-button hack entirely:

```python
st.markdown('<div class="category-card-btn">', unsafe_allow_html=True)
if st.button("Therapy Area", key="go_ta", use_container_width=True):
    ...
st.markdown('</div>', unsafe_allow_html=True)
```

Note: The `st.markdown` open/close div trick does NOT wrap Streamlit widgets. So instead:

**ACTUAL RECOMMENDED APPROACH:** Override button styles specifically inside bordered containers on the landing page. Add CSS:
```css
/* Category card buttons - override pill styling */
[data-testid="stVerticalBlockBorderWrapper"] .stButton > button {
    border-radius: var(--radius-lg) !important;
    font-size: 16px !important;
    font-weight: 600 !important;
    padding: 20px 16px !important;
    min-height: 60px !important;
    white-space: normal !important;
    text-align: left !important;
}
```

This targets buttons inside bordered containers (which only exist on the landing page) and makes them look card-like.

#### Step 2: Fix Home Navigation — Single Unified Approach (P0 #2)

**The core problem:** The SVG icon in the HTML header cannot trigger `st.session_state` changes. You need a Streamlit widget for internal navigation. Currently there are TWO elements: a decorative SVG icon (non-functional) + a "Home" button in the breadcrumb.

**Solution — Remove the SVG icon, keep only the breadcrumb button:**
- Remove `home_icon_html` from `header.py` entirely
- The breadcrumb "Home" button on sub-pages IS the navigation. Style it to look like a clean text link (not a pill button). This is standard breadcrumb UX.
- Add CSS to target the breadcrumb buttons specifically to look like text links rather than pill buttons.

**ALTERNATIVE Solution — Make the header itself the clickable element:**
- Remove the SVG icon
- Move the Streamlit "Home" button INTO a columns layout that sits AT the same level as the header markdown, positioned to the far right
- Use CSS negative margin to pull it up into the header area visually
- This is fragile and breaks on different screen sizes

**RECOMMENDED:** Keep the breadcrumb approach (it's a standard pattern) but:
1. Remove the SVG icon from the header (it's misleading)
2. Style the "Home" breadcrumb button as a subtle text-link (underline on hover, no pill border)
3. Add specific CSS for `.breadcrumb-home` class
